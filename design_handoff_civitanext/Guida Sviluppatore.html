sviluppatore

Tutto il necessario per trasformare il prototipo HTML in un prodotto reale: direzione di design, architettura dei file di riferimento, modello dati e la sequenza di sviluppo consigliata per dipendenze.

I file del bundle sono riferimenti di design, non codice di produzione. Sono prototipi in HTML/React+Babel che mostrano aspetto e comportamento desiderati. Vanno ricostruiti nel codebase di destinazione con i suoi pattern e librerie, non copiati così come sono. Auth, dati e persistenza sono finti (utenti hardcoded, array statici, localStorage): vanno sostituiti con auth reale, database e backend.

Direzione di design

Stile "collage editoriale": carta crema con texture, accenti corallo, grafiche fatte a mano (starburst a raggi, onde, sottolineature ondulate), ombre dure (offset, mai sfocate), niente gradienti, niente emoji. Alta fedeltà: colori, tipografia e spaziature sono definitivi.

  • Ombra dura 5px 5px 0 rgba(30,58,95,0.14), nessun blur.
  • Grafiche brand (starburst, onde, logo) generate in civitanext-ui.jsx: riutilizzabili come componenti SVG.

Roadmap per dipendenze

L'ordine è per dipendenze tecniche e valore consegnabile, non per peso visivo. Ogni fase produce qualcosa di usabile e testabile. Principio: prima le fondamenta (auth + dati + una feature verticale completa), poi le feature una alla volta riusando le stesse fondamenta.

Fase 0 Fondamenta tecniche 1-2 settimane

  1. Stack: Next.js (App Router) + React + TypeScript; DB Postgres (Supabase/Neon) + Prisma.
  2. Setup repo, lint/format, CI base.
  3. Design system: token in CSS/Tailwind; componenti base e grafiche SVG.
  4. Schema DB iniziale: User, Event, Thread, Reply, Proposal.

Sblocca tutto il resto.

Fase 1 MVP autenticato 3-4 settimane

  1. Auth reale, ruoli socio/admin (login separati nel comportamento).
  2. Iscrizione a 3 passi (pagamento simulato finché non serve Stripe).
  3. Profilo socio: tessera, preferenze su DB.
  4. Eventi (lettura + RSVP): prima feature verticale completa.
  5. Forum: thread, risposte, voto unico.

Esce un prodotto già usabile. Dipende da Fase 0.

Fase 2 Partecipazione civica 3-4 settimane

  1. Proposte e votazioni (form soci, stati, voto unico).
  2. Coda di approvazione admin: ciclo revisione → votazione → approvata.
  3. Pannello admin: panoramica, proposte, moderazione, gestione eventi.
  4. Quiz: motore domande/risposte + gestione admin.
  5. Attuazione proposte (feedback loop).

Il cuore della missione. Dipende da auth (Fase 1).

Fase 3 Mobile e PWA 2-3 settimane

  1. Responsive o shell mobile dedicata; rimuovere il toggle iOS/Android del prototipo.
  2. PWA installabile, cache offline, service worker.
  3. Notifiche in-app, poi push (Web Push).

Ha senso dopo le feature core (Fase 1-2).

Fase 4 Community e città 3-4 settimane

  1. Città: mappa (Leaflet/Mapbox), spazi, galleria (upload reale), documenti (PDF), timeline.
  2. Community: mentorship, competenze, webinar (video), rassegna stampa.
  3. Reputazione e badge (calcolano sui dati delle feature precedenti, quindi per ultimi).
  4. Sondaggi rapidi, sync Google Calendar reale, email digest (cron + Resend/SendGrid).

Alto valore, non bloccanti. Riusano i pattern già costruiti.

Fase 5 Analytics, testing, deploy continuo

  1. Analytics admin (engagement, retention, trend): query aggregate + grafici.
  2. Testing: unit (voti, auth, stati proposte) + e2e (Playwright) sui casi di Flusso di Test.html.
  3. Deploy: Vercel/Firebase, DB gestito, storage immagini, env per le chiavi.
  4. Hardening: rate limiting, validazione server-side, GDPR, backup.

Dipendenze in breve

Fase 0 (fondamenta)
   --> Fase 1 (auth, profilo, eventi, forum)
          --> Fase 2 (proposte, admin, quiz, attuazione)
                 --> Fase 3 (mobile / PWA / notifiche)
                 --> Fase 4 (community, citta, reputazione, email)
                        --> Fase 5 (analytics, testing, deploy)

Note pratiche. Non costruire tutte le UI prima del backend: una feature alla volta, verticale (DB → API → UI). La timeline storica ha voci "da completare": è raccolta contenuti con tempi lunghi non tecnici, avviarla presto in parallelo. Flusso di Test.html è insieme guida di QA manuale e mappa dei casi e2e: tenerlo aggiornato.
